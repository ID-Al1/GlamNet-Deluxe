## 0. CRITICAL SCOPE FINDING — read this before anything else

**The live web app's frontend source code is not in the GitHub repo.**

Evidence:
- Repo top level is only: `.agents/`, `artifacts/`, `attached_assets/` (2 PDFs), `lib/`, `scripts/` + config files. There is **no `client/`, `web/`, or `src/` for the live site** (checked; `/client` 404s).
- `artifacts/api-server/src/app.ts` mounts **only `/api`** — it serves no static frontend.
- `artifacts/mobile/` is an Expo (React Native) app — but the live site uses web-DOM libraries (react-day-picker calendar, hamburger drawer, footer links), so the live site is *not* the Expo web export.
- `artifacts/mockup-sandbox/` is a design sandbox with state-based navigation (no router, no auth) — it is *not* what's deployed at glam-net.replit.app (live site has `/login`, `/signup`, real signups like "Test Placer" / "Alwande Khoza", live DB IDs like `sp_demo001`).

**Conclusion:** the live frontend lives only inside the Replit workspace (the `glam-net` Replit app), which has never been pushed to this GitHub repo. **The implementation session must run inside Replit (or the live client must be pushed to GitHub first).** Everything in `mockup-sandbox/` is a *reference design*, not the production code. Do not "beautify the mockups" and call it done — that was the disconnected-prototype trap this brief exists to prevent.

**First action of the implementation session:** open the Replit workspace, locate the web client (likely `client/` or `src/` with React + shadcn/Tailwind — the live UI shows shadcn-style calendar and theme toggle), and map its routes to this checklist before touching anything.

## 1. Settled decisions (fixed direction — do not reopen)

1. **Tagline:** "Beauty. Your Way."
2. **Palette (a blend):** cream base background, black for text and hero-image scrims, **gold as the accent** (sparkle icons, verified badge, highlights), existing brand **rose #C4526E demoted to small secondary highlights only**. Saved tokens for reference: rose `#C4526E`, gold `#B8893A`, cream `#F9F6F2`.
   - Repo state today (`mockup-sandbox/src/glamnet/colors.ts`): LIGHT theme uses canvas `#FAF7F4`, charcoal `#1C1A19`, accent **terracotta `#B8765C`**, taupe `#E8E1DA`, sage `#7C8B6F`, gold `#B8893A` (defined but barely used). DARK theme uses rose `#C4526E` as accent. `theme.ts` has a second, dark-only token set (`bg #15100C`, rose, gold). **Neither matches the decided palette** — the live site meanwhile leans rose/pink for primary buttons. A single token pass (cream/black/gold, rose secondary) is needed and should be the first visual change, since everything else inherits from it.
3. **Solo vs studio profiles — activity-based tiering.** Profile richness scales with real activity: newer/less active artists get a lighter profile (fewer stats, no team section, no business-hours panel); more completed bookings + longer tenure unlocks the fuller studio-style profile automatically. Whatever tier renders, **the completed-bookings count is always genuinely displayed** — it's the trust signal driving the tiering.
   - **Flag: this is a genuinely new mechanic, not a tweak.** Nothing in the repo supports it: `types.ts` `Artist` has no `completedBookings`, `responseTime`, `memberSince`, `languages`, or team fields; no threshold rule exists anywhere. It needs: (a) a computed activity metric from the real `bookings` table (which exists in `lib/db`; roadmap Stage 1 adds `agreedRate`/`talentRole`/`city`), (b) a threshold rule (e.g. `completedBookings >= N && tenureDays >= M` unlocks the rich profile), (c) two profile layout variants. Design it as its own task; do not bundle into the visual pass.

## 5. Guardrail audit — violations found in the repo (fix during the pass)

1. **Fake faces:** `glamnet/data.ts` gives all 32 mock artists `randomuser.me` portrait URLs — stock/AI-adjacent faces presented as GlamNet artists. Violates guardrail #3. Live site correctly shows no fake faces (but shows broken-empty image areas instead). **Remove randomuser portraits from any code path that could ship; replace with initials/brand-texture placeholders.**
2. **Fake stats:** `data.ts` `PLATFORM_STATS` (380 professionals, 2 400 bookings, 47 brands) and invented ratings/review counts throughout mock data. Fine in a sandbox, must never render on the live app.
3. **"0 (0)" ratings on live** instead of "New artist" (guardrail #2 / brief Screen 2 item 4).
4. **No shimmer loading placeholders** anywhere on live — hero and card image areas render as flat empty blocks (guardrail #6).
5. SA locations: **compliant** — both live and mock data are fully South African. No US placeholders found.
6. Minor code bug to fix if touching `mockup-sandbox/ArtistProfile.tsx`: the Book CTA style contains a malformed string — `fontFamily: "'Inter', sans-serif', marginTop: 8"` (the marginTop is trapped inside the font string).
